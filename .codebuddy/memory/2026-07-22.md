### 关键架构决策（2026-07-14 ~ 07-22 会话）
- **抓取主流程 `scrapeAllProducts`**（src/app/producthunt.content/index.tsx）：
  - `RESOLVE_REDIRECT` 串行，每 2s 一个（避免触发 Cloudflare）
  - `SCRAPE_EMAILS` 最多 5 个 worker 并行，不阻塞解析
  - 解析失败 → `OPEN_TAB` 兜底，不抓、不记入 `resolvedUrls`
- **`RESOLVE_REDIRECT`（src/app/background/index.ts）**：最多重试 3 次，每次失败等 1s。必须以「URL 真正离开 `producthunt.com/r/`」为完成条件（PH /r/ 由 Cloudflare 保护，中转页会先 complete）。超时 25s 返回 null。
- **email-scraper.ts `attemptFetch`**：已修改为 `!response.ok`（含 403）时也 fallback 到 `fetchWithBrowserTab`（浏览器标签带登录态过 Cloudflare），否则直接 SCRAPE_EMAILS(/r/) 必 403 抓不到。
- **`fetchProductUrls`**：GraphQL `posts(first: 50)`，`pageSize=50` 已是 PH API 上限（不能再大）。每页后等 1s；遇 429 等 5s 重试（page--; continue）。
- **`openAllProductTabs` / reopen**：改为 5 个 worker 并发，去掉 1s sleep。
- **`resolvedUrls`**：当前仍是 `string[]`，reopen 用 `resolvedUrls.length > 0 ? resolvedUrls : urls`。之前讨论过改 Map 以全量覆盖（已解析用真实地址、未解析回退 /r/），但未实施。
